# 🤖 B站长效登录状态维护模块（Cookie Auto-Refresh）开发指南





### Step 1: 状态检查 (Check Status)
- **动作**：调用 B站的 [Cookie 状态查询接口]。
- **逻辑分支**：
  - 若返回 JSON 中指示 `refresh: false`：说明当前 Cookie 极度健康，**立即终止任务，不做任何操作**（严禁高频无效刷新）。
  - 若返回 JSON 中指示 `refresh: true`：说明进入了缓冲期，进入 Step 2。
  - 若接口报错提示未登录：说明 Cookie 已失效（硬过期或被风控踢出），触发异常告警，终止任务并等待人工重新登录。

### Step 2: 发起刷新 (Execute Refresh)
- **动作**：组装当前有效的凭证，调用 [Cookie 刷新接口]。
- **必要参数准备**：
  - 提取当前的 `SESSDATA`、`bili_jct` (CSRF Token)。
  - 提取当前保存的 `Refresh Token`。
  - **动态加密**：调用 B站 [实时 RSA 公钥获取接口]，利用获取到的公钥对上述信息及时间戳进行加密签名（具体加密规则请检索最新规范）。
- **预期结果**：获取到全新的 `SESSDATA`、全新的 `bili_jct` 以及 **全新的 `Refresh Token`**。

### Step 3: 确认刷新 (Confirm Refresh) 【⚠️ 关键易错点】
- **动作**：在拿到新凭证后，必须立即调用 [刷新确认接口]。
- **参数**：携带旧的 `Refresh Token` 和新的 CSRF Token。
- **目的**：告知 B站服务端新凭证已签收，注销旧凭证。若跳过此步，账号会被风控拉黑。

### Step 4: 持久化存储 (Persistence)
- **动作**：覆写本地存储（数据库/配置文件）。
- **要求**：必须同时保存最新的 `SESSDATA`、`bili_jct` 和 **最新的 `Refresh Token`**。若遗漏更新 `Refresh Token`，下一次刷新将永远失败。

## 🔍 API 接口寻址指南 (需 Agent 自行检索)

B站的 API 及加密签名算法（如 Wbi 签名、RSA 加密逻辑）可能会随时间更新。**本指南不硬编码具体的 URL。** 在编写实际网络请求代码前，请你（Agent）利用搜索能力或查阅最新的开源文档（如 `bilibili-API-collect` 项目），定位并实现以下具体接口及算法：

1. **获取 Cookie 状态接口** (通常包含 `/x/passport-login/web/cookie/info`)
2. **获取 Web 登录 RSA 公钥接口**
3. **刷新 Cookie 接口** (通常需要复杂的加密 Payload)
4. **确认刷新 Cookie 接口** (通常包含 `confirm` 关键字)
5. **Wbi 签名算法** (当前 B站高频使用的接口防刷签名机制，若刷新流程需要则必须实现)

## 🚫 绝对不可触犯的风控红线（防御性设计）

在编写代码时，必须强制加入以下限制：
1. **频率限制**：Step 1 的检查频率不得高于 `1次/12小时`。严禁写成死循环或高频轮询。
2. **请求伪装**：所有向 B站发起的 HTTP 请求，必须携带合理的且**固定**的 `User-Agent`，建议实现随机延迟机制（Sleep）避免定时任务的并发特征。
3. **异常捕获**：任何网络超时、接口 500 报错、返回非预期 JSON 时，必须捕获异常并跳过本次流程，保留旧 Cookie，切勿在报错状态下清除数据库凭证或疯狂重试。
